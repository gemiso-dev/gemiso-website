# 프로젝트 규칙

## 개발 서버

- 개발자가 `npm run dev`로 개발 서버를 직접 띄워 둔 상태로 작업한다. (기본 포트 3000)
- **임의로 별도의 dev 서버를 실행하지 말 것.** `npm run dev`, `next dev` 등을 새로 띄우거나, 포트를 바꿔(`-p 3001` 등) 중복 실행하지 않는다.
- 변경 사항 확인이 필요하면 이미 떠 있는 서버(`http://localhost:3000`)를 사용한다.
- 개발자가 띄운 서버를 임의로 중지하지 않는다. 서버 중지·재시작은 개발자가 명시적으로 요청할 때만 한다.

## 빌드

- **`npm run dev`가 동작 중일 때는 `npm run build`를 실행하지 말 것.** dev 서버와 `npm run build`가 같은 `.next` 폴더를 공유해, 빌드가 dev 서버 상태를 깨뜨려 500 오류를 일으킨다.
- 타입·컴파일 검증이 필요하면 빌드 대신 `npx tsc --noEmit`처럼 `.next`를 건드리지 않는 방법을 쓴다.
- 정적 빌드(`npm run build`) 검증이 꼭 필요하면, 먼저 개발자에게 dev 서버를 잠시 멈춰도 되는지 확인한 뒤 실행한다.

## 연혁

- 연혁 데이터는 두 곳에 나뉘어 있다. 새 소식을 추가·수정할 때는 **둘 다** 갱신한다.
  - [components/history-data.ts](../../components/history-data.ts) — `/history`에 쓰는 **큐레이션된 주요 이정표**(시대별로 강조할 항목만).
  - [components/history-detail-data.ts](../../components/history-detail-data.ts) — `/history/detail`에 쓰는 **연도·월 단위 전체 실적**.
- 추가 절차:
  1. 상세(`history-detail-data.ts`)에는 모든 실적을 해당 연도 블록에 **빠짐없이** 넣는다. 새 연도가 시작되면 연도 블록을 배열 맨 끝에 추가한다.
  2. 그중 강조할 이정표(설립·특허·인증·핵심 제품·주요 방송사/기관 첫 도입·전략적 제휴 등)만 골라 큐레이션(`history-data.ts`)의 해당 시대 `items`에도 추가한다.
- 두 파일 모두 데이터는 **연대순(오래된 → 최신)**으로 두고, 페이지에서 역순으로 렌더링한다. 같은 항목의 연·월 표기와 순서가 두 파일에서 어긋나지 않게 맞춘다.
- 정리: **상세에 없는 연혁이 큐레이션에만 있어서는 안 된다.** 큐레이션에 항목을 넣었으면 상세에도 반드시 반영한다.
